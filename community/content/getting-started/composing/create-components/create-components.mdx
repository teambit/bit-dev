# Creating components

Bit makes it simple to build independent [components](/components/component-overview) and compose them with others, using [dependencies](/dependencies/dependencies-overview).  
Each component may have be of a different type, like Node Modules, React Components, Angular Modules and more. The type is defined according to the [Component Development Environment (Env)](/getting-started/composing/dev-environments) you set for a component.

Bit has several supported Envs, like [Node](/frameworks/node/node-overview), [React](/frameworks/react/react-overview), [Angular](/frameworks/angular/angular-overview) and [LitHTML](/frameworks/lit-html/lit-overview). <!-- Envs can be found in the [Bit Cloud community hub](/). -->  
Each env comes with a set of pre-configured templates for components. Transforming the process of creating components to be simple and repeatable with best practices at its core. See available templates by running the `bit templates` command in your workspace.

```bash
bit templates
```

```bash
teambit.react/react
    react (a basic react component)
    react-context (a react context component)
    react-hook (a react hook component)
    react-env (customize the base React env with your configs and tools)

teambit.harmony/node
    node-env (customize the base Node env with your configs and tools)
    module (a general purpose node module)

teambit.mdx/mdx
    mdx (Content component)
```

:::info Registered templates

The list of templates is populated either with pre-sets from Bit or with any [custom templates](/dev-services/generator/generate-component) you created.

:::

Create a component with the `create` command. Here we'll create a simple `react` component named `my-welcome`, which will be in the `ui` namespace.

```bash
bit create react ui/my-welcome
```

```bash
1 component(s) were created

company.demo/ui/my-welcome
    location: base-ui/ui/my-welcome
    env:      teambit.react/react
```

That's it! Bit has generated a new component. The component has its own [directory](/workspace/structure/component-directory).  
From now on Bit treats this component as an independent module, which so should you.

:::tip Fine-tune control for generating components

By default, components are created according to the `defaultScope` defined in the [workspace.json](/workspace/workspace-configuration), and the `env` defined as part of the template.  
Override any of them by passing the `--scope` and `--env` flags to the `create` command. Change the location of the component by passing the `--path` flag.

:::

## Component directory

Let's view the newly created component directory. This is an example of a basic React component. If you chose any other template, your output may be slightly different.

```bash
└── ui/my-welcome                  # Component directory
    ├── index.ts                   # Main component file.
    ├── my-welcome.tsx             # Implementation of the component.
    ├── my-welcome.spec.tsx        # Test file of the component.
    ├── my-welcome.composition.tsx # Compositions for simulating the component in different states.
    └── my-welcome.docs.mdx        # Docs file for the component.
```

As seen above, the created component has few different files generated by default.

- `index.ts` - [Component main file](/components/structure/component-main-file), exposing the component API.
- `my-welcome.tsx` - Component's implementation. This can be one, or many files.
- `my-welcome.spec.tsx` - [Tests](/dev-services/tester/tester-overview) file for the Component.
- `my-welcome.compositions.tsx` - [Compositions](/dev-services/compositions/compositions-overview) file to simulate the component in different states.
- `my-welcome.docs.md` - [Documentation](/dev-services/docs/docs-overview) file. A markdown file for documenting the component.

You can decide to remove the `*.spec.*`, `*.docs.*` or `*.composition.*` files, as they are not mandatory. We recommend using them, as it makes it easier for other developers to understand how to use your component and simplify maintaining it in the long run.

## Check component status

Whenever we create a new component, or done any modification for a component, we recommend checking the [workspace status](/workspace/workspace-status).

```bash
bit status
```

