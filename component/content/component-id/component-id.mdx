# Component ID

import { Image } from '@teambit/docs.ui.zoomable-image';
import componentTree from './component-tree.png';
import componentID from './component-id.png';

Component ID is the unique identifier of a component. It is a combination of a the component's **scope** and **name**.

<Image src={componentID} width="50%" />

* **Scope** - defines the owner of the component.
* **Name** - describes the component's goal. Supports nested namespaces.

Component ID is set the moment you add or create a component.

## Get component ID

There are several commands and options to get a complete view of all available components and their IDs.

1. `bit list` locally, or with the `--remote` option to see all components in a remote scope.
1. Run `bit start` in a workspace, or browse the remote scope UI.

## Referring to components

Anytime you want to run a command or get information on a component, use the component ID as an identifier for it. This is supported in various commands:

```bash
bit show teambit.community/ui/graph/bubble-graph
bit dependents teambit.community/ui/graph/bubble-graph
bit tag teambit.community/ui/graph/bubble-graph
```

Through all outputs Bit uses the component ID to refer to a component.

## Scope name and hosting

Scope describe the owner of the component on one hand, on the other hand, scope ownership is also defined by "who has access to the component", and what's the access level.  
Read/write access for components is defined in a [remote scope](/scope/running-a-scope-server), so by setting the ownership of a component, you also define where it will be hosted.

### Pre-fix scopes with the `.` separator

Bit supports pre-fixing for scopes with an organization name for creating ownership on scopes. [bit.cloud](https://bit.cloud) use this to have better control over naming with distributing components the community hub.

All scopes created in [bit.cloud](https://bit.cloud] follow this pattern for the scope name:

```bash
ACCOUNT.SCOPE
```

## Workspace component layout

Bit structures your workspace according to the component ID. It destructure the ID and create a directory tree. This makes the workspace easier to navigate by creating symmetry between the logical structure of components and their physical location.

When creating new components you can define the component's scope with the following syntax:

```sh
bit create react-component forms/add-product
```

Bit will create the component in the following directory:

```sh
.
└── inventory
    └── forms
        └── add-product
```

## Component module name

All components in a workspace are available for consumption as node modules. Bit does it by creating a module for each component in `node_modules` according to it's component ID. However, as Bit heavily utilizes scoping and namespaces, it needs to transform component IDs to module names Node can resolve.

Let's take the following component ID as an example:

```sh title="Component ID with an organization pre-fix"
company.scope/forms/add-product
```

Bit generates the following module name:

```sh title="Organization prefix transforms to npm scope, name is separated by ."
@company/scope.forms.add-product
```

Use the result module name in all your `import * from ...` statements.

:::tip Generate module links

Bit generates module links in `node_module` dir for all components when creating them and during `bit install`. To trigger this action run `bit link`.

:::

## Rename components

Component ID is used in `import` statement of its dependents, this means that updating it breaks consumers. To workaround that Bit essentially build a new component and deprecate/remove the previous one.

```sh
bit rename teambit.base-ui/button teambit.advanced-ui/button
```

When you have the right ID:

* Run `bit dependents <old-component-id>` to get a list of dependent components.
* Refactor all `import {...} from <old-module-name>` to `import {...} from <new-module-name>` for dependent components.
* Now your components all use the right component.

## Component name limitations

Bit Components must be require-able by node module resolution. The component ID must fit the same naming constraints for modules in `node_modules` directory:

* Component name length should be greater than zero.
* All the characters in the component name must be lowercase.
* Component name can consist of hyphens.
* Component name must contain only url-safe characters.
* Component name should not contain any leading or trailing spaces.
* Component name should not contain any of the following characters: `~)('!*` .
* Component name cannot be the same as a node.js/io.js core module nor a reserved/blacklisted name.
* Component name length cannot exceed 214.
